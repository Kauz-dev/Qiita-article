## はじめに

2025年、AIはもはや「未来の技術」ではなく、日常のインフラになりつつあります。
コードを書くとき、文章を作るとき、画像を生成するとき——気がつけばAIが隣にいる時代になりました。

この記事では、現在のLLM（大規模言語モデル）を取り巻く状況を整理し、エンジニア視点で「今どう付き合うか」を考えてみます。

---

## LLMの現在地

### 主要モデルの勢力図

2025年時点で注目されている主なLLMは以下のとおりです。

| モデル | 提供元 | 特徴 |
|---|---|---|
| GPT-4o / o3 | OpenAI | 汎用性が高く、ツール連携が強力 |
| Claude 3.x / 4.x | Anthropic | 長文理解・安全性重視 |
| Gemini 2.x | Google DeepMind | マルチモーダル性能が高い |
| Llama 3.x | Meta | オープンソース、ローカル運用可能 |
| Qwen / Mistral など | 各社 | 特定用途・軽量モデルとして台頭 |

かつては「ChatGPT一強」という印象でしたが、今は用途ごとに使い分ける時代になっています。

## エンジニアとLLMの付き合い方

### コーディングアシスタントとして

GitHub Copilot、Cursor、Claude Codeなどのツールが普及し、コードの補完・生成・レビューがAIによって高速化されています。

特に最近のモデルは：

- 関数単位でなく、**ファイル・プロジェクト全体を把握した補完**が可能
- バグの原因特定や修正案の提示
- テストコードの自動生成

といった場面で実用レベルに達しています。

```python
# 例：LLMにコードレビューを依頼するイメージ（Anthropic SDK）
import anthropic

client = anthropic.Anthropic()

message = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[
        {
            "role": "user",
            "content": "以下のPythonコードをレビューしてください:\n\ndef add(a, b):\n    return a + b"
        }
    ]
)
print(message.content[0].text)
```

### RAG（検索拡張生成）との組み合わせ

LLM単体では知識のカットオフや事実誤認（ハルシネーション）の問題があります。
これを補う手法として **RAG（Retrieval-Augmented Generation）** が広く使われています。

仕組みは単純で：
1. ユーザーの質問に関連するドキュメントをベクトル検索で取得
2. そのドキュメントをコンテキストとしてLLMに渡す
3. LLMは「取得した情報を元に」回答する

社内ドキュメントへの質問応答や、最新情報を扱うチャットボットなどに活用されています。

## AIを使う上で意識したいこと

### ハルシネーションへの対処

LLMは「もっともらしい嘘」をつくことがあります。特に：

- 存在しないライブラリや関数名を生成する
- 古い情報を最新情報として提示する
- 数値計算を誤る

といったケースは今でも起こります。**生成された内容を鵜呑みにせず、必ず検証する習慣**が重要です。

### プロンプトエンジニアリング

モデルの性能を引き出すには、プロンプトの書き方が重要です。

- **役割を与える**：「あなたはシニアエンジニアです」
- **出力形式を指定する**：「JSON形式で返してください」
- **Chain of Thought**：「ステップバイステップで考えてください」

シンプルな工夫ですが、回答の品質が大きく変わります。
